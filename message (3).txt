2. Exp B is conceptually broken as written

This is the biggest remaining flaw.

You define clean examples as Bucket 1 and corrupted examples as Bucket 2, then patch the Bucket 1 hidden state into the Bucket 2 forward pass. The document says this tests whether patching the Bucket 1 latent state into the Bucket 2 example recovers the correct answer.

That is not clean causal patching.

Why? Because the Bucket 1 and Bucket 2 examples are different problems with different correct answers. Even if they are nearest-neighbor matched, injecting the latent state from one problem into another can easily be semantically nonsensical. If it fails, you learn almost nothing. If it works, reviewers may ask whether you injected answer-like information from another example.

Fix it like this:

Clean run: same example under vanilla LatentMAS
Corrupt run: same example under W_a removed, KV blocked, shuffled KV, or top-k removed
Patch: restore one latent site from clean into corrupted
Metric: recovery of the same example's correct-answer logit/probability

That is much cleaner. Patching should usually be same-example clean/corrupt, not different-example Bucket 1 → Bucket 2 transfer.

You can keep matched-pair patching as exploratory, but it should not be “the only mechanism proof.”

3. The document contradicts itself on what is core

It says CORE 8 includes L, B, M, P, Q, D, C, H.

C, D, and H are not core. They are support/appendix unless they directly change the main claim.

The actual core is baselines + communication isolation + subspace intervention + W_a/KV ablation + probe/gating.

Activation patching itself is not run — only the pair files are generated.
